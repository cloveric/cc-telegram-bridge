# Codex Telegram-Out File Delivery Design

## Goal

Add a Codex-only file return path that does not depend on the model understanding Telegram transport. When a user asks a Codex-backed bot to generate a file and send it back, Codex should write that file into a request-scoped output directory, and the bridge should send the file to Telegram after the turn completes.

## Context

Current behavior is unreliable for Codex. Prompt-level instructions that describe Telegram file delivery are treated as ordinary conversation context, and Codex frequently falls back to local workspace writes or refuses the request outright. Claude may be guided by prompt/instruction merging, but Codex `exec` is optimized for task execution, not conversational capability inference. This makes file delivery a product capability problem, not a prompting problem.

The existing bridge already supports sending files to Telegram when a model returns a fenced block in the format:

```text
```file:filename.ext
<file contents>
```
```

That path is useful for small text outputs, but it is still model-driven. This design adds a second, platform-driven path for Codex: request-scoped output directory capture and automatic `sendDocument`.

## Scope

### In Scope

- Codex-backed instances only
- Request-scoped output directory creation under instance workspace
- System-level instruction telling Codex where to place returnable files
- Post-turn directory scan and Telegram `sendDocument`
- Safe limits on file count and size
- Tests for empty output, single-file output, and multi-file output

### Out of Scope

- Claude engine changes
- Streaming file delivery during generation
- Recursive directory upload as a zip
- Rich UI for browsing generated files
- Prompt-based file transfer as the primary Codex path

## Approaches Considered

### 1. Prompt-Only File Delivery

Tell Codex in prompt text that Telegram can send files if it uses a fenced `file:` block.

Rejected. This is already proven unreliable in practice. Codex treats the capability text as ordinary conversation context and frequently ignores it or actively rejects it as non-authoritative.

### 2. Scan Entire Workspace For New Files

Record the workspace state before a turn, diff it after the turn, and send every new file.

Rejected. This is unsafe and noisy. It risks leaking unrelated files, cache artifacts, or stale files from prior runs. It also makes file-send intent ambiguous.

### 3. Request-Scoped Telegram-Out Directory

Create a per-request output directory and instruct Codex to place returnable files there. After the turn completes, only scan that directory and send those files.

Chosen. It matches Codex's local-file workflow, provides a clear safety boundary, and makes file delivery deterministic.

## Selected Design

### Request-Scoped Output Directory

For each Codex request, create a directory like:

```text
<instance-state-dir>\workspace\.telegram-out\<request-id>\
```

Where `<request-id>` is generated by the bridge for that specific Telegram request. The directory is empty before the turn begins.

### Instruction Injection Model

For Codex only, inject a system-level instruction block that says:

- If the user wants a file sent back through Telegram, write the final file into the provided output directory
- Only place files intended for Telegram delivery in that directory
- Prefer UTF-8 text/code files for small generated artifacts
- Do not place temporary scratch files there

This instruction is not framed as a "Telegram capability" claim. It is framed as a workspace/output contract:

- "Files written to this directory will be returned to the user after the task completes"

That is much closer to Codex's execution model than chat transport language.

### Post-Turn Collection

After the Codex turn completes:

1. Scan the request-scoped output directory
2. Filter to regular files only
3. Apply limits:
   - Maximum file count per turn
   - Maximum per-file size
   - Maximum total bytes sent
4. Send each file using Telegram `sendDocument`
5. If the model also returned text, keep the existing text response path

### No-Output Behavior

If the output directory is empty:

- Do nothing extra
- Preserve the normal text-only response flow

## High-Level Data Flow

```text
Telegram message
  -> access check
  -> session lookup
  -> if engine == codex:
       create request output dir
       inject codex output-dir instruction
       run adapter
       collect files from request output dir
       send files via Telegram
       send / edit final text response
  -> if engine != codex:
       preserve existing behavior
```

## Required Changes

### Bridge Layer

The bridge should create or carry a request-scoped output directory only for Codex-backed requests. This should be passed to the adapter as part of request instructions or a new field on the adapter input.

### Delivery Layer

After adapter completion for Codex requests:

- inspect the request output directory
- send files before or alongside final response
- record audit metadata for sent files

### Adapter Layer

Codex adapters should merge the instance `agent.md` and the per-request output-directory contract cleanly so that existing instance personality remains intact.

### Utility Layer

Add a focused helper module to:

- create request output directories
- enumerate produced files
- enforce file-count/size limits
- cleanly describe file-send results

## Safety Constraints

- Only scan `.telegram-out/<request-id>/`
- Never scan the full workspace
- Ignore directories, hidden scratch content, and symlinks if encountered
- Reject files that exceed configured limits
- If multiple files are generated, send them in deterministic sorted order
- Keep the mechanism Codex-only in Phase 1

## User Experience

If Codex generates files in the output directory:

- Telegram receives those files directly
- The textual reply may still explain what was generated

If Codex generates no file:

- User receives only the standard textual reply

If Codex tries to write outside the output directory:

- The file is not auto-sent
- No special handling is performed in Phase 1

## Error Handling

- Directory creation failure: fail the request early with a clear error
- File scan failure after successful turn: keep text response, log audit error, and surface a concise Telegram warning if needed
- `sendDocument` failure: do not suppress it silently; return a visible error path similar to existing message-send failures

## Testing Strategy

### Unit / Integration Cases

- Codex request with empty output directory -> no file sent
- Codex request with one new file in output directory -> exactly one `sendDocument`
- Codex request with multiple files -> deterministic send order
- Non-Codex request -> output-directory logic not engaged
- Existing text response path still works when files are also sent
- Output directory contract is merged with instance instructions, not replacing them

### Regression Guarantees

- Existing fenced `file:` block support remains unchanged
- Claude file behavior is unaffected
- No workspace-wide file scan is introduced

## Success Criteria

- Codex-backed bot can return at least one generated file through Telegram without relying on conversational capability inference
- The mechanism is constrained to request-scoped output directories
- Existing Codex and Claude text flows continue to pass all tests
